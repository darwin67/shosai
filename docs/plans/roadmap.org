* Overview

Shōsai (書斎, "study/reading room") is a desktop ebook reader supporting EPUB,
PDF, and CBZ formats, built in Rust with a native GUI.

** Goals

- Fast, native desktop ebook reader
- Support for EPUB, PDF, and CBZ/CBR formats
- Clean library management for book collections
- Customizable reading experience (fonts, themes, layout)
- Permissively licensed (Apache-2.0)
- Potential future mobile support

** Licensing Decisions

- *MuPDF ruled out*: AGPL-3.0 copyleft would infect the entire application.
- *Poppler ruled out*: GPL-2.0 via the C library, despite MIT Rust bindings.
- *~epub~ crate ruled out*: GPL-3.0 licensed.
- *PDFium selected*: Apache-2.0 licensed, battle-tested (used in Chromium).
- *EPUB parser*: Custom implementation to avoid all GPL dependencies.

*Note — PDFium bundling*: The app depends on ~libpdfium~ at runtime (a C
shared library). For development this is handled by the Nix dev shell
(~pdfium-binaries~ in ~flake.nix~), but every distribution format must
solve this independently:
- *Nix package*: Add ~pdfium-binaries~ as a runtime dependency and patch
  ~RPATH~ / wrap with ~LD_LIBRARY_PATH~.
- *AppImage / Flatpak*: Bundle ~libpdfium.so~ alongside the binary inside
  the container.
- *macOS .app*: Embed ~libpdfium.dylib~ in the bundle's ~Frameworks/~
  directory and set ~@rpath~ accordingly.
- *Windows*: Ship ~pdfium.dll~ next to the ~.exe~ (DLL search order finds it
  automatically).
- Alternatively, investigate ~pdfium-render~'s ~static~ feature or building
  PDFium from source as a static library to eliminate the shared-library
  dependency entirely.
- The ~bblanchon/pdfium-binaries~ GitHub repo provides pre-built binaries
  for all major platforms that CI can download during the release build.

* Key Technical Risks & Mitigations

| Risk                                        | Impact | Mitigation                                                       |
|---------------------------------------------+--------+------------------------------------------------------------------|
| Iced is "experimental" software             | High   | Pin to 0.14, abstract GUI layer, monitor upstream closely        |
| PDFium binary distribution complexity       | Medium | Nix handles dev; each distro format must bundle libpdfium       |
| No pure-Rust HTML/CSS renderer for EPUB     | High   | Simplified renderer for MVP, webview fallback in Phase 5         |
| Iced text rendering quality on non-HiDPI    | Medium | Test early, may need custom text rendering via canvas            |
| Iced slow startup on Linux (font loading)   | Low    | Known issue #2455, may be fixed upstream; can lazy-load fonts    |
| ~pdfium-render~ types thread safety         | Medium | Investigate early, may need message-passing architecture         |
| Complex PDFs rendering incorrectly          | Medium | PDFium is battle-tested (Chromium), low risk in practice         |
| EPUB content diversity (tables, math, etc.) | Medium | Simplified renderer won't handle all content; phase 5 addresses  |

* Research Notes

** MuPDF (ruled out)
- Crate: ~mupdf~ v0.6.0 on crates.io (repo: ~mupdf-rs~ on GitHub)
- Supports PDF, EPUB, CBZ, SVG, HTML, XPS natively
- Renders to pixmaps with raw RGBA access
- *License: AGPL-3.0* — copyleft, would require entire app to be AGPL
- Types are ~!Send + !Sync~
- Would have been the simplest solution (one library for all formats)

** PDFium (selected)
- Crate: ~pdfium-render~ v0.8.37
- Apache-2.0 / MIT dual license
- Rust wrapper around Google's PDFium (used in Chromium)
- Full PDF page rendering to pixel buffers
- Requires bundling ~libpdfium~ shared library at runtime
- Actively maintained, 84 releases

** EPUB format
- EPUB is a ZIP archive containing XHTML, CSS, images, and XML metadata
- Structure: ~META-INF/container.xml~ → OPF file → spine + manifest
- Simple enough to parse with ~zip~ + ~roxmltree~ (~300 lines)
- All existing Rust EPUB parser crates are GPL-licensed

** CBZ format
- CBZ is a ZIP archive containing images (JPEG/PNG), sorted by filename
- Trivial to implement with ~zip~ + ~image~ crates
- CBR uses RAR (restrictive license for extraction library)
- Recommend supporting CBZ only initially

** Iced GUI framework
- v0.14, self-described as "experimental"
- Supports ~Handle::Rgba~ for displaying raw pixel data — critical for our use case
- Has ~Canvas~ widget for custom 2D drawing
- No built-in webview widget
- No built-in virtualized list (use ~Scrollable~ + ~Sensor~ + ~Lazy~)
- File dialogs via external ~rfd~ crate
- Known issues: slow Linux startup (#2455), text rendering on non-HiDPI (#2254)
